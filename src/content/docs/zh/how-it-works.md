# 工作原理

邻传是一款**点对点**应用：没有中央服务器，同 LAN 上的设备直接相互通信。本页是 v0.5.1 的技术总览。

## 分层结构

```
┌─────────────────────────────────────────────┐
│ UI                                           │
│   桌面端：Vue 3 + Tauri (Rust)              │
│   移动端：Flutter + Rust FFI                │
├─────────────────────────────────────────────┤
│ 服务层（Rust）                              │
│   API · 诊断 · 缩略图缓存                   │
├─────────────────────────────────────────────┤
│ 核心（Rust）                                │
│   身份 · 发现 · 配对                         │
│   会话 · 传输 · 存储                         │
└─────────────────────────────────────────────┘

网络: UDP 发现 + 自适应单播 ↔ TLS 1.3 会话连接池
存储: SQLite (AES-256-GCM) + 本地文件系统
```

## 发现

启动后应用每 2 秒发出一次 **UDP 广播**，包含设备 id、名称、平台、传输端口、证书指纹。约 15 秒未收到对端广播则将其标记为离线。

如果路由器/防火墙拦截广播，邻传会回退到 **自适应单播**——继续向已知 peer 发送单播包，让已配对设备保持可见。

## 身份与加密

- 每台设备首次启动时生成一对 **密钥与自签名证书**。
- 设备身份就是该证书的 **SHA-256 指纹**。
- 所有会话使用 **TLS 1.3 + 双向证书认证（mTLS）**，基于 [rustls](https://github.com/rustls/rustls)。
- 加密 **强制开启**——没有降级路径。

配对会锁定对端指纹。后续连接会在每次 TLS 握手时重新校验，指纹不一致即拒绝，可阻止"被攻陷的局域网"上的中间人冒充。

## 会话连接池（v0.5.1）

早期版本每个文件开一条 TCP；v0.4.1 改为 **会话级连接池**，v0.5.1 进一步加固：

- 每条会话维护 **4–8 条 TLS 长连接 slot**。
- ≤ 50 MB 文件占一条 slot 跑完。
- > 50 MB 大文件分片到最多 **4 条 slot 并行**。
- 单条会话最多 **5 个文件并发**。
- 发送端无法主动连对端时（严格对称 NAT、防火墙），会发出 **反向 slot 请求**让对端回连，沿用同一会话。
- v0.5.1 起，slot 中途断开时会以 250 ms / 1 s / 4 s ± 抖动指数退避，最多重连 3 次。

## 文件传输

文件按 **2 MB 分块** 传输。每收到一个 chunk，接收端严格按以下顺序：

1. 写入 chunk 数据
2. 对数据文件 `fsync`
3. 更新 manifest（`Acked`）
4. 对 manifest `fsync`
5. 发送 ACK

这一顺序保证：突然掉电不会出现"已完成但内容残缺"的文件。连接中断后下次会从最后一个 `Acked` 的 chunk 继续，复用 partial 文件与 manifest。

## 接收端加固（v0.5.1）

- **路径遍历守卫** —— 接收端拒绝绝对路径、`..` 段、NUL 字节、Windows 保留名。恶意 peer 无法写到接收目录之外。
- **原子去重** —— 两份同名文件并发到达时，使用 `O_EXCL` 占位避免静默覆盖；第二份文件自动加数字后缀。
- **取消即时响应** —— 点取消的退出延迟从 30+ 秒降到亚秒级。

## 本地存储

- 会话与消息历史保存在 **本地 SQLite 数据库**，使用 **AES-256-GCM** 加密。
- 接收的文件以明文形式保存在 `~/Downloads/Link2Send/`（可配置），**不会自动加密**——请妥善保管设备。
- 应用 **不会发起任何外网连接**：无遥测 SDK、无崩溃上报、无推送通道。
